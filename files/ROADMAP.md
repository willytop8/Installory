# ROADMAP.md

Phased delivery plan. Each phase has a definition of done; we don't move to the next until the current one's checklist is satisfied.

## Phase 0: Foundation (week 1)

**Goal:** A buildable Swift project that can persist to SQLite, request folder access from the user, and render a placeholder window. No real scanning yet.

**Deliverables:**

- [ ] Xcode project initialized for sandboxed macOS app, App Store distribution profile
- [ ] `Package.swift` with GRDB as the only SPM dependency
- [ ] Sandbox entitlements file: `com.apple.security.app-sandbox` + `com.apple.security.files.user-selected.read-only`
- [ ] `FolderAccessManager` — wraps NSOpenPanel and security-scoped bookmarks
- [ ] `PathDiscovery` implementation covering Apple Silicon and Intel Homebrew prefixes
- [ ] `Database` setup with GRDB DatabasePool and a no-op migration
- [ ] Empty `ContentView` showing "Cruft" with the app icon
- [ ] App launches, doesn't crash, and writes a stub row to SQLite on first run

**Why this is a separate phase:** the sandbox and folder-access machinery has to be solid before scanners come online. If access is shaky, every downstream scanner is shaky.

## Phase 1: Brew-only inventory (week 2)

**Goal:** A working Mac app that scans Homebrew and shows you what's there, with descriptions.

**Deliverables:**

- [ ] `PackageScanner` protocol defined
- [ ] `BrewScanner` walks `<prefix>/Cellar/<formula>/<version>/INSTALL_RECEIPT.json` and `<prefix>/Caskroom/<cask>/<version>/INSTALL_RECEIPT.json` directly — no `brew` invocation
- [ ] User grants Cruft read access to `/opt/homebrew` (or `/usr/local`) via NSOpenPanel on first launch
- [ ] `ScanCoordinator` with `TaskGroup` orchestration (even though there's only one scanner; we need the structure)
- [ ] `InventoryView` shows a table of packages with name, version, installed date, description
- [ ] **Bundled descriptions corpus** for ~7,000 brew formulae and casks, generated by `scripts/generate-descriptions/` from `formulae.brew.sh`
- [ ] Search box that filters the list
- [ ] Click a row → detail pane with full info (dependencies, install path, exact install time)
- [ ] First-launch flow: scan progresses with a per-manager status row

**Out of scope:** cleanup, snapshots, provenance.

**Definition of done:** William can run the app on his own Mac, see his brew packages with descriptions, search them, and click into detail. No actions yet.

## Phase 2: Multi-manager inventory + safety primitives (weeks 3–4)

**Goal:** Pip and npm scanners working alongside brew, plus snapshot infrastructure (without uninstall yet).

**Deliverables:**

- [ ] `PythonInterpreterDiscovery` enumerating all Python interpreters
- [ ] `PipScanner` reading `dist-info` directories directly (no `pip list` subprocess for parsing)
- [ ] System Python detection — `isReadOnly: true` for any package in `/usr/bin/python3` or CLT paths
- [ ] `NpmScanner` for plain npm globals (pnpm/bun deferred to later)
- [ ] Per-manager scanner status surfaced in UI honestly: "brew: 247 · pip: 89 across 3 interpreters · npm: 12 · pnpm: skipped"
- [ ] `Package.manager` shown as a colored badge per row
- [ ] Filter by manager in the UI
- [ ] `SnapshotManager` can capture a full snapshot to disk as JSON
- [ ] Settings pane: list snapshots, view a snapshot's contents (no restore yet)

**Definition of done:** The app gives William a unified view of every package across brew, pip (all interpreters), and npm on his Mac. He can take a snapshot but not yet restore or remove anything. **This is v0 by your definition.**

## Phase 3: Cleanup script generation (week 5)

**Goal:** The user can multi-select packages and receive a clean, copy-pasteable shell script that, when they run it themselves in Terminal, removes those packages.

**Deliverables:**

- [ ] `ScriptGenerator` produces a shell script per multi-package selection: dependency-aware ordering within each manager, `set -e` header, verbose `echo` of each command before it runs
- [ ] Per-manager uninstall commands documented and tested against fixtures (brew, pip per-interpreter, npm globals, pipx, cargo, gem)
- [ ] Cleanup wizard UI: multi-select packages → script preview pane → "Copy to clipboard" + "Save as .sh" buttons
- [ ] Snapshot captured automatically before the script is generated (the snapshot is the export manifest the user can use to restore later)
- [ ] System Python and other read-only packages are filtered out of the generation step entirely (greyed out with explanation in the UI)
- [ ] Common-essentials denylist (`git`, `curl`, `openssl`, `python@*`, etc.) is rendered as commented-out lines with a `# WARNING:` banner the user must uncomment to actually run

**Definition of done:** William selects packages, sees the script, copies it into Terminal, and watches it run with full visibility. Cruft itself never executed anything.

## Phase 4: Provenance (weeks 6–7)

**Goal:** "When did I install this and what was I doing?" works for the Claude Code user.

**Deliverables:**

- [ ] `ProvenanceCollector` reads:
  - filesystem mtimes (brew `INSTALL_RECEIPT.json`, pip `dist-info` mtimes, etc.)
  - `~/.zsh_history`, `~/.bash_history` for install commands (user-granted folder access)
  - `~/.claude/projects/*/sessions-index.json` and the per-session JSONL transcripts (user-granted folder access)
- [ ] Evidence is stored structured in SQLite (not as freeform text)
- [ ] `NarrativeRenderer` produces template-based human-readable text from structured evidence (Swift string templates; no LLM, no API)
- [ ] Detail pane shows: install time (with confidence label), nearby activity, narrative
- [ ] User can click through to "see the evidence" — the raw signals we used

**Definition of done:** William opens the app, clicks a brew formula he installed three months ago, and reads back exactly what project he was working on and what other things he installed that day.

## Phase 5: Polish for first App Store release (weeks 8–9)

**Goal:** Cruft is ready for a small TestFlight beta and then App Store submission.

**Deliverables:**

- [ ] Onboarding flow — first launch, brief explanation, asks the user to grant read access to each package manager directory (brew prefix, ~/.cargo, ~/.pyenv, etc.) and optionally to ~/.zsh_history and ~/.claude
- [ ] App Store Connect set up: bundle ID, screenshots, marketing copy, privacy nutrition label
- [ ] TestFlight beta build and tester group
- [ ] App Store Review prep: written explanation of why we need file-access entitlements, with the legitimate use case documented for the reviewer
- [ ] About box with version, build, attribution
- [ ] Settings: snapshot management, manager toggles, folder access management
- [ ] Crash handling: surface errors gracefully, never crash silently
- [ ] Performance pass: scan completes in <15s on a populated Mac
- [ ] Memory pass: idle app under 200MB

**Definition of done:** Cruft is in TestFlight, five testers install and use it, and we're ready to push the App Store Submit button.

## Future (after first release)

In priority order, roughly:

1. **More managers:** cargo, gem, mas, pipx as first-class. pnpm, bun, yarn classic.
2. **Conda / mamba** — high-value for ML users, ugly to scan.
3. **Update advisor** — outdated packages with changelogs.
4. **Security advisories** — `pip-audit` / `npm audit` integration (still as generated scripts).
5. **"Explain this stack"** — clustering packages that work together.
6. **Disk space reclaimed** estimates.
7. **Brewfile / requirements.txt export** for new-machine setup.
8. **Linux support** — only if there's demand.
9. **Power-user mode toggle** — show raw shell commands by default, show dependency graphs, etc.
10. **Direct-download non-sandboxed version** — for users who want Cruft to perform the uninstall itself rather than generating a script. Would ship as a notarized DMG with Developer ID signing, outside the App Store, sandbox off. Same codebase, different distribution target. Only worth doing if real demand emerges.

## What this roadmap is not

- A commitment to dates. Weeks are estimates; reality will shift them.
- A scope contract. If Phase 3 reveals that uninstall should be in Phase 2, we'll merge.
- A list of features to maximize. We are explicitly *not* trying to match GlazePKG's 36 package managers. We want to do brew + pip + npm exceptionally well first.
