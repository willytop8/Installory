# Safety

The hard rules for anything that touches the user's packages. Read every time you touch a code path that generates a cleanup script. No exceptions.

## The premise

The user is afraid of breaking their machine. Installory has to be safer than doing nothing. A single bad uninstall — even a polite-sounding one that breaks Python — and we've lost trust for that user permanently.

The defining design choice for v1: **Installory never executes anything that changes the user's system.** When the user wants to clean up, Installory generates a shell script. The user reads it, runs it themselves in Terminal, and watches each command execute with full visibility. The destructive moment lives in the user's hands, in the tool they already trust for irreversible work.

## The five rules

### Rule 1: Snapshot behavior is preference-driven for per-package removal; always-on for batch

Installory supports a `snapshotBeforeRemoval` preference (Always / Ask each time / Never) that governs per-package cleanup scripts. Batch cleanup always captures a snapshot regardless of this preference.

When a snapshot is requested and succeeds, the cleanup sheet shows a green "Snapshot captured" confirmation. When a requested snapshot *fails*, the sheet shows a prominent red warning — this state is distinct from "user chose to skip a snapshot." The UI must never silently present a failed snapshot as a successful skip.

Batch cleanup code path:

```swift
await generateAndShowCleanupScript(packages: selected, captureSnapshot: true)
```

Per-package code path routes through `requestRemoval(_:)` which resolves the preference and passes `captureSnapshot: true/false` accordingly. A `Never` preference is a legitimate user choice, not a safety violation.

### Rule 2: Generated scripts are defensive by default

Every generated script starts with the same preamble:

```bash
#!/usr/bin/env bash
# Cleanup script generated by Installory on 2026-05-15 at 14:23.
# Snapshot captured before generation: <snapshot-id>.
# Review every line before running. Installory did not run this script.

set -euo pipefail
```

`set -e` aborts on the first failed command. `set -u` catches missing variables. `set -o pipefail` propagates pipe failures. Each command is also preceded by a `echo "→ <command>"` line so the user sees what's about to run in real time.

### Rule 3: The user runs the script; Installory does not

There is no "execute" button. Installory surfaces the generated script in a code-styled pane, with two actions: **Copy to clipboard** and **Save as .sh**. The UI tells the user exactly what to do next: open Terminal, paste, and press Enter — or `chmod +x` the saved file and run it. We don't `osascript` Terminal open, we don't drag-drop into the user's shell, and we don't pretend to be any kind of runner. The destructive moment is the user's, deliberately.

### Rule 4: Read-only packages never appear in a generated script

System Python and other read-only packages (anything under `/usr/bin`, `/Library/Developer/CommandLineTools/`, or `/Applications/Xcode.app/Contents/Developer/`) are filtered out before the script is generated. In the UI, they're greyed out with an explanation tooltip and cannot be selected in the multi-select.

### Rule 5: Common essentials become commented-out lines with a warning

Some packages are technically removable but removing them is almost always wrong. The denylist (`git`, `curl`, `openssl`, `python@*`, `node`, etc.) does not get filtered out — but each matching package is rendered into the script as a *commented* line with a warning header the user must consciously uncomment:

```bash
# ─────────────────────────────────────────────────────────────────────────
# WARNING: the following packages are common essentials. Most systems
# and many other packages depend on them. They are commented out below.
# Uncomment a line ONLY if you are certain you do not need it.
# ─────────────────────────────────────────────────────────────────────────
# brew uninstall git
# brew uninstall openssl@3
```

This preserves user agency — a power user with a real reason can still remove `git` — while making the default action a no-op.

## Read-only packages

A package is read-only if any of these are true:

- It's installed under `/usr/bin`, `/Library/Developer/CommandLineTools/`, or `/Applications/Xcode.app/Contents/Developer/`
- It's a Python package inside a system Python (per `python-problem.md`)
- It's depended on by another read-only package

The `isReadOnly: Bool` flag on `Package` is computed during scan and persisted. The script generator refuses to include any package with `isReadOnly == true`.

### Common essentials denylist

```
brew: git, curl, wget, openssl, openssl@3, ca-certificates, gnupg, gpg, python@*, node, ffmpeg, sqlite
pip:  (interpreter-level: pip, setuptools, wheel)
npm:  npm, corepack
```

The denylist is a static JSON file in the bundle, easy to edit. Add to it when we find a class of mistakes users make.

## Snapshot mechanics

### What's in a snapshot

A snapshot is a single JSON file (stored as a row in `snapshots`) containing:

- Every package across the managers in scope (see scoping note below)
- Each package's name, version, qualifier (for pip), and `isExplicit` flag
- The timestamp and a free-text note
- The reason (`manual` | `preCleanup` | `autoFirstScan`)

### Snapshot scoping

Pre-cleanup snapshots intentionally capture only the packages being removed, not the full inventory. The snapshot is the restore point for *that specific cleanup operation*, so its scope matches the generated script. The restore diff ("Restore Missing Packages") then tells you exactly which of those packages are gone.

First-scan and manual snapshots capture the full live inventory.

### Snapshot kinds

Installory captures three kinds of snapshots:

- **First-scan snapshot** (`autoFirstScan`) — automatically, the first time the user scans. A one-time baseline. Controlled by the `backshelf.firstScanSnapshotTaken` UserDefaults flag (prefix retained from prior product name).
- **Pre-cleanup snapshot** (`preCleanup`) — automatically before batch cleanup scripts. Per-package removal respects the `snapshotBeforeRemoval` preference (Always / Ask / Never).
- **Manual snapshot** (`manual`) — user-initiated, from the toolbar or the menu.

No automatic deletion. The user can delete snapshots manually in Settings. Disk impact is small (<200KB per snapshot for a populated Mac, JSON-compressed).

### Restoring missing packages from a snapshot

The Snapshots screen's primary action is **"Restore Missing Packages…"**. Installory diffs the snapshot against the current live inventory and shows only the packages that are no longer installed — not the whole snapshot. The user reviews the checklist, unchecks anything they don't want, and clicks "Generate Reinstall Script". Installory produces a shell script the user runs themselves in Terminal — same model as cleanup, in reverse.

The diff matches on `(manager, qualifier, name)` — not version. The question is "is this package present at all?", not "is this exact version still installed?"

```
Snapshot from 2026-05-14:
  brew: ffmpeg, openssl, ...
  pip: requests, ...

Current state:
  brew: openssl, ... (ffmpeg is gone)
  pip: requests, ...

Diff result: [ffmpeg is missing]

Generated reinstall script:
  #!/usr/bin/env bash
  set -euo pipefail
  # snapshot recorded 7.0.1; Homebrew installs the current version
  echo "→ brew install ffmpeg"
  brew install ffmpeg
```

**The script is additive only:** it adds back what's missing; it never `uninstall`s anything installed since the snapshot.

**Whole-snapshot reinstall is explicitly rejected.** A snapshot can hold 200+ packages most of which are still installed; a whole-snapshot script is noise and risks re-installing things the user removed on purpose. The feature is always a diff.

**Version fidelity is honest per manager:**
- pip / npm / pipx / cargo / gem: pin the exact recorded version.
- brew / brewCask: cannot pin an arbitrary historical version; installs the current version and emits an inline comment naming the recorded version.
- mas: no CLI reinstall path; emits a comment directing the user to the App Store.

### Snapshots are not transactional

If a user's reinstall script involves 30 install commands and the 15th fails, `set -e` stops execution. The user is now in a partial state and can choose what to do next — re-run the script (subsequent installs will skip what's present), edit the script, or move on. We do not pretend snapshots are atomic transactions. They're a record so you have something to restore from, not a guarantee of rollback.

## Dependency awareness

The `ScriptGenerator` computes a removal order using the scanned dependency graph:

1. Build a graph of `<package> requires <package>` from the inventory.
2. For each selected package, find its dependents.
3. If a dependent is not selected for removal, emit a `# WARNING:` comment above the uninstall line: "X depends on Y. Removing Y will break X."
4. Order the selected packages by topological sort so leaves are removed before roots within each manager.

For dependencies across managers (e.g., a brew formula required by a Python package), we don't try to model. We only do dependency analysis within a single manager.

**Known limitation:** dependency warnings consider only the packages in the current selection, not the full inventory. A package that depends on a selected package but is not itself visible in the current filter may be silently omitted from warnings.

## Never do

These are never available, regardless of user request:

- Execute any command on behalf of the user. Anything destructive happens in their Terminal, by their hand.
- Emit `sudo` into a generated script. If a removal would need root, we comment the line out with a note: "This package's removal requires sudo; we don't suggest sudo'd commands. If you really want this, run it yourself."
- Emit `rm -rf` into a generated script. We use the manager's own uninstall command.
- Touch `/usr/bin`, `/System`, or any system directory at any level — even read.

## Testing safety code

Safety code gets extra test rigor:

- **Script-shape tests.** For every selection scenario (read-only filtered, denylist commented, dependency-ordered, cross-interpreter pip), assert the generated script matches a fixture exactly. These are pure-function tests against the script generator — no filesystem mutation, no real shell.
- **Snapshot round-trip tests.** Capture, serialize, deserialize, verify equality.
- **Read-only check.** Table-driven test covering every path category.
- **Denylist coverage.** For every denylist entry, verify that the generator emits it as a commented line and not as an active command.

Because Installory never executes a cleanup, we have no integration tests against real shell-outs. The script is the artifact; testing the script's shape is testing the safety guarantee.

## When safety conflicts with helpfulness

When in doubt: choose safety. A user who wanted to clean up and had to read the script before running it will still be a user tomorrow. A user whose Python broke will not.
