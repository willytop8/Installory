# Installory — Implementation Handoff

**Audience:** a Claude Code CLI agent acting as **orchestrator**, delegating to subagents.
**Goal:** implement nine improvements to Installory's organization features, grounded in the existing architecture, without violating any of the product's safety guarantees.

This document is the entry point. Read it fully, then read every spec in `handoff/specs/` before assigning work. The specs are written to be handed to subagents more or less verbatim.

---

## 1. What Installory is (so subagents don't have to re-derive it)

Installory is a sandboxed, read-only macOS app that inventories packages across eight managers (Homebrew formula + cask, pip, pipx, npm, Cargo, RubyGems, Mac App Store) and helps users understand and safely clean up their machine. It is aimed at people — especially new users of AI coding tools — who have accumulated packages they don't understand.

**Non-negotiable product guarantees. Every change must preserve all of these:**

- **Read-only.** The app inspects; it never modifies the system. All scanning is `FileManager` reads through user-granted, security-scoped bookmarks. **No subprocess execution.** (PATH resolution in spec 01 reads `ProcessInfo.environment`, not a shell-out — see that spec.)
- **No network.** Zero network calls at runtime. No new dependencies that phone home. Descriptions are bundled.
- **Never deletes, never executes.** Cleanup is *script generation* the user runs themselves in Terminal. No feature here may delete a package, write to a package directory, or execute a generated script.
- **Sandboxed.** App Sandbox is on with `files.user-selected.read-only`. Any feature that needs to read a path outside an existing grant (notably provenance, spec 08) must route through the user-granted folder-access flow — it cannot assume ambient filesystem access.

If a spec ever appears to require breaking one of these, **stop and surface it** rather than working around it.

## 2. Architecture you will be working in

Two layers, split deliberately:

- **`Installory/Sources/InstalloryCore/`** — a pure, UI-free Swift package. All real logic lives here: scanners, models, cleanup/script generation, provenance collectors, snapshots, persistence (GRDB), descriptions, export. **This is where the 336 tests live and where all new logic belongs.**
- **`App/Sources/`** — thin SwiftUI shell. `AppCoordinator` (`@Observable @MainActor`) holds all cross-screen state; `Views/` render it. Keep views dumb.

**Architectural rule for this whole effort:** new analysis logic (PATH resolution, severity scoring, orphan detection, staleness scoring, AI-session matching, report rendering) goes in `InstalloryCore` as pure, `Sendable`, unit-tested functions/types. The App layer only wires results to the screen. If you find yourself putting `if`-logic in a SwiftUI `View`, move it down into Core.

**Build / project facts that will bite you if ignored:**

- The Xcode project is generated by XcodeGen from `project.yml` and is **gitignored**. New files under `App/Sources/` only enter the Xcode build after `./scripts/regenerate-xcode.sh`. New files under `Installory/Sources/` and `Installory/Tests/` are picked up automatically by SwiftPM (no regen needed for `swift test`).
- `App/Sources` is a **recursive** path in `project.yml`, so new subfolders are fine — just regenerate.
- Library tests run with no Xcode: `cd Installory && swift test`. Swift 6 strict concurrency is on; everything crossing actor boundaries must be `Sendable`.
- Only SPM dependency is GRDB. Do not add dependencies without flagging it.

## 4. Subagent orchestration strategy

Recommended division into **four subagents**, run in two waves. Each subagent owns a disjoint set of *primary* files; shared files are serialized through the orchestrator (section 5).

**Wave 1 (parallel):**

- **Subagent A — Duplicates cluster (specs 01, 02, 03).** Owns `Models/DuplicateGroup.swift`, `Views/DuplicatesView.swift`. Does all three specs in order in one branch so the shared edits to those two files don't conflict with themselves.
- **Subagent B — Insights (specs 04, 05).** Owns new Core analysis files (orphans, staleness) + their views. Independent logic.
- **Subagent C — Provenance (spec 08).** The largest, highest-risk workstream — start it early. Owns `Provenance/` wiring, `Persistence/ProvenanceDAO` integration, `PrivacyTab`, provenance display in `PackageDetailView`.

**Wave 2 (after Wave 1 merges):**

- **Subagent D — Reports & history (specs 06, 07) + AI-session tagging (spec 09).** Specs 06/07 are independent and could run in Wave 1 if you have capacity; 09 must wait for 08. Grouped here because all three are "present existing data in a new way" and low-conflict.

Use the `general-purpose` subagent type for implementation work and the `Explore` type for any read-only reconnaissance. For the final verification gate (section 6) spawn a **separate** subagent so review is independent of the author.

If you prefer fewer agents, the safe serial order is: **08 → 01 → 02 → 03 → 04 → 05 → 06 → 07 → 09.**

## 5. Shared "hot" files — serialize edits to these

Several files are touched by multiple workstreams. **Do not let two subagents edit these concurrently.** Apply edits one workstream at a time and rebase the others:

- `App/Sources/AppCoordinator.swift` — new computed properties / actions for nearly every spec.
- `App/Sources/Models/SidebarSelection.swift` *and* `Installory/Sources/InstalloryCore/Models/SidebarSelection.swift` — specs 03, 04, 05 add sidebar cases. **Note both copies exist** (Core is the real enum; App has a stub/typealias). New cases go in the **Core** enum; keep its `userDefaultsKey` / `init?(userDefaultsKey:)` / `filtered(by:)` switches exhaustive (no `default:`).
- `App/Sources/Views/SidebarView.swift` — new sidebar rows.
- `App/Sources/Views/RootView.swift` — routes `sidebarSelection` → content view; every new pane adds a case.
- `App/Sources/Extensions/PackageManager+Display.swift` — symbols/colors for any new badges.
- `App/Sources/Views/SettingsView.swift` — specs 07 (export buttons) and 08 (consent toggle) both add sections.
- `Installory/Sources/InstalloryCore/Demo/DemoData.swift` — every spec adds demo fixtures; serialize all additions through the post-wave coordination pass.

A clean approach: have each subagent implement its **Core logic + tests first** (these rarely conflict), merge those, then apply the thin App-layer wiring for all specs in a single coordinated pass to the hot files.

## 6. Global verification protocol (the definition of done)

Per your instruction, verification is a single global gate rather than per-feature acceptance criteria — but it is mandatory and runs after each workstream merges, and once more at the very end.

Run, from repo root unless noted:

1. **Library tests must stay green and grow.** `cd Installory && swift test`. The suite is at 336 tests today; every workstream that adds Core logic must add tests, and the count must only go up. A red suite blocks the merge.
2. **Strict-concurrency clean.** The build must compile with **zero warnings** under Swift 6 (the project builds warning-free today — keep it that way).
3. **App builds, both configs.** `./scripts/regenerate-xcode.sh` (required if any `App/Sources/**` files were added), then `xcodebuild -project Installory.xcodeproj -scheme Installory -configuration Debug build` and `-configuration Release build`. Unsigned build is fine for verification.
4. **Demo mode still exercises every feature.** Each new view must render from `DemoData` (`AppCoordinator.enterDemoMode()`), since App Review runs on a clean device with no package managers. If a feature has no demo representation, extend `Installory/Sources/InstalloryCore/Demo/DemoData.swift` so it does. Launch with `-demo` (or `INSTALLORY_DEMO=1`) and confirm.
5. **Guarantees audit.** Grep the diff for `Process(`, `URLSession`, `FileManager` writes, `.write(`, network symbols. Confirm no subprocess, no network, no writes to package directories, no script execution. Provenance (08) must default **off** and read nothing until the user opts in *and* grants access.
6. **Docs updated.** `README.md` "What it does" / "Project status" and the design docs under `files/` reflect shipped features (notably: provenance moves from "in development" to shipped once 08 lands; duplicate section gains PATH/severity language).

Spawn a fresh subagent for a final independent pass over items 1–6 before declaring done.
